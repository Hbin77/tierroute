# Native dense-ridge protocol

Status: experimental training-only protocol. It is not used by the default wheel,
quickstart, routing CLI, or dependency-free reference solver.

The optional native process solves one already encoded centered-ridge problem. It does
not extract features, standardize columns, choose folds, fit calibrators, or tune policy
parameters; those boundaries stay under Python orchestration. The reviewed project C
source contains no networking operation, but that source-review fact is not an
attestation about arbitrary caller-selected executable bytes.

## Transport contract

The caller supplies an already-open regular request file as standard input and captures
the standard-output response through a bounded binary pipe into an owner-only regular
file. Both byte streams are fixed-order, little-endian binary. Text, JSON, native C
structs, platform-sized integers, padding, and trailing bytes are forbidden. Every
count is unsigned and every numeric cell is an IEEE-754 binary64 value.

### Request version 1

| Offset | Width | Field | Required value |
|---:|---:|---|---|
| 0 | 8 | magic | ASCII `TRRIDG01` |
| 8 | 4 | version | `1` as `uint32` |
| 12 | 4 | flags | `0` as `uint32` |
| 16 | 32 | request ID | opaque caller-generated bytes |
| 48 | 8 | sample count `n` | positive `uint64` |
| 56 | 8 | feature count `d` | positive `uint64` |
| 64 | 8 | target count `m` | positive `uint64` |
| 72 | 8 | ridge | finite, positive binary64 |
| 80 | `8*n*d` | feature matrix `X` | row-major finite binary64 |
| next | `8*n*m` | target matrix `Y` | row-major finite binary64 |

The file must end after the last target. The implementation checks every multiplication
and addition used to derive payload, allocation, output, and work sizes before allocating
or reading a dense matrix.

### Reviewed version-1 ceilings

| Resource | Maximum |
|---|---:|
| samples `n` | 1,000,000 |
| features `d` | 4,096 |
| targets `m` | 256 |
| sidecar executable bytes | 16 MiB |
| request bytes | 4 GiB |
| response bytes | 16 MiB |
| C heap estimate | 2 GiB |
| deterministic work units | 32,000,000,000 |
| captured standard error | 16 KiB plus a local truncation marker |
| child-process timeout | 600 seconds; default 60 seconds |

The mirrored Python/C work estimate is

```text
3*n*(d+m) + n*d*(d+1)/2 + n*d*m + d^3 + 2*m*d^2 + m*d
```

and the heap estimate covers `X + Y + 2*d*d + 2*m*d + 2*d + 3*m` binary64
cells. These are fail-fast audited ceilings, not a promise that a small host can safely
run a request near them. A deployment may impose smaller limits but must never raise
them without a new resource review.

### Response version 1

| Offset | Width | Field | Required value |
|---:|---:|---|---|
| 0 | 8 | magic | ASCII `TRRRES01` |
| 8 | 4 | version | `1` as `uint32` |
| 12 | 4 | status | code below |
| 16 | 32 | request ID | exact request echo |
| 48 | 8 | feature count `d` | exact request echo when parsed |
| 56 | 8 | target count `m` | exact request echo when parsed |
| 64 | `8*m` | intercepts | success only, target order |
| next | `8*m*d` | weights | success only, target-major |

An error response ends after its 64-byte header. A success response has status zero,
finite coefficients, and no trailing bytes.

| Status | Meaning |
|---:|---|
| 0 | success |
| 1 | malformed header, payload, version, flags, or exact-length contract |
| 2 | count, checked-arithmetic, allocation-size, or reviewed-work bound refused |
| 3 | non-finite numeric input or invalid ridge |
| 4 | memory allocation failed |
| 5 | factorization, output, or residual verification failed |
| 6 | I/O, platform, or internal contract failed |

Status numbers are protocol data, not process exit codes. The process also exits
nonzero on error. The caller first authenticates and validates any complete structured
response, then cross-checks it with the process exit: success requires both exit zero
and status zero, while a known error requires both a nonzero exit and a header-only
nonzero status. Missing, contradictory, or malformed evidence fails closed. Standard
output contains only the response. Standard error is diagnostic-only, bounded by the
caller, and never parsed as protocol data.

## Numerical contract

For `X` with shape `(n, d)`, `Y` with shape `(n, m)`, and positive `ridge`, the process
centers each column, then solves

```text
(X_centered^T X_centered + ridge I) W = X_centered^T Y_centered
```

One positive-definite Cholesky factor is shared across all `m` targets. Intercepts are
recovered as `mean(Y) - mean(X) W`. The process rejects non-finite intermediates. For
each target it also checks the normal-equation residual `r = A w - b` using

```text
||r||inf <= 4096 * DBL_EPSILON * (d + 1)
            * max(1, ||A||inf * ||w||inf + ||b||inf)
```

where `A = X_centered^T X_centered + ridge I`; vector infinity norms are maximum
absolute entries and the matrix infinity norm is the maximum absolute row sum. Norms,
residuals, and their scale are accumulated as C `long double`, which may have the same
precision as `double` on a conforming target. This is a corruption/solver-failure
guard, not the parity criterion. The implementation does not promise coefficients that
are byte-identical to the Python reference or between CPU/compiler platforms. Parity
is evaluated separately with coefficient and prediction tolerances, while a
platform-local binary hash authenticates the executable itself.

The committed parity fixtures use the following `pytest.approx` gates; either the
relative or absolute allowance may satisfy a scalar. They are regression thresholds,
not universal forward-error guarantees for every conditioning regime.

| Fixture | Weights | Intercepts | Fitted predictions |
|---|---|---|---|
| analytic/multi-target/collinear modest matrices | `rel=1e-8`, `abs=1e-9` | `rel=1e-9`, `abs=1e-10` | `rel=1e-9`, `abs=1e-9` |
| near-singular matrix, `ridge=1e-7` | `rel=1e-6`, `abs=1e-8` | `rel=1e-7`, `abs=1e-8` | `rel=1e-7`, `abs=1e-8` |
| unprojected 1,024-feature dense matrix | `rel=1e-6`, `abs=1e-8` | `rel=1e-7`, `abs=1e-8` | `rel=1e-7`, `abs=1e-8` |

Ill-conditioned systems can have materially different coefficients while preserving
predictions, so both coefficient and prediction gates are mandatory. A failed gate is
never waived merely because the normal-equation residual passed.

Version 1 receives already encoded features. In particular, the first three surface
columns have already been standardized by the training-only `PromptFeatureSchema`; a
native process must never standardize them again.

## Process boundary

The Python caller must receive an explicit absolute executable path and expected
lowercase SHA-256. Paths beginning with `//` or `\\` are rejected on every platform to
exclude UNC and device-style spellings. A mapped drive or mounted network filesystem
cannot be detected portably, so the caller must separately establish that every path is
local. The digest authenticates only the caller-selected bytes; it is not executable
approval, source provenance, an import audit, or proof of network absence.

Before embedding materialization, training preflight authenticates a nonempty regular
non-symbolic-link executable no larger than 16 MiB through an open descriptor. The solve
path repeats the size and SHA-256 checks while producing its private executable snapshot,
so replacing the configured path after preflight cannot change the bytes that execute.
The adapter runs with `shell=False`, a restricted environment and working directory, and
enforces child timeout, stderr, response-size, identity, shape, status, and exact-length
limits. Runtime PATH discovery and automatic executable download are forbidden. The
timeout clock begins after child launch; pre-authentication filesystem reads and bounded
request serialization occur before that clock and have byte ceilings rather than an
end-to-end wall-clock deadline.

Matrix inputs are caller-owned sized sequences and must remain unchanged for the whole
`solve` call. The adapter deliberately avoids a second full Python object snapshot at
the 1,036-feature target scale; concurrent mutation or custom sequence side effects are
unsupported and can be rejected during serialization. Project training paths pass
immutable tuple snapshots.

The base wheel contains no native executable and keeps `dependencies = []`. Native
source is included only in the source distribution. Any released executable needs a
separate platform hash, link/import audit, license record, SBOM entry, and offline test.

## Deliberate non-goals

This dense protocol alone does not make full nested LODO efficient. The experimental
[prepared graph contract](prepared-session-graph.md) now fixes the unique subset/score
target and modeled numeric-resource admission preflight, but it does not execute that
graph. A bounded [in-memory reference](prepared-feature-store.md) now accepts
caller-precomputed prompt embeddings and leakage-tests training-only domain-statistic
combination, but it does not run a provider, solve, or score the graph. A later scalable
prepared training session must authenticate offline embedding preparation, consume the
reference moments, solve coefficients, and batch raw prediction scores. That
higher-level protocol needs a separate session/training-engine identity, magic, and
version distinct from the ridge-solver identity, plus end-to-end parity and resource
proof; it must not overload this single-problem format.
